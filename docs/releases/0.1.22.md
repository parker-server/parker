# Parker 0.1.22 Release Notes

Status: Draft

## Added

- Added a real volume-level `Following` workflow so users can track future issues of a specific run without implicitly subscribing to every historical volume in a series.
  Product model: `Following` behaves like a comic-shop subscription for a specific volume. The user follows a run, Parker uses that follow timestamp as the baseline, and only later qualifying issues surface as new arrivals.
  Scope note: `Following` is volume-based because Parker's data model treats volumes as the actual run-level container, while a top-level series can span many unrelated historical volumes.

- Added the first user-facing `Following` surfaces across the app.
  Entry points: Users can now follow and unfollow directly from the volume detail page.
  Discovery surface: Home now includes a `New from Following` rail for newly arrived mainline issues from followed volumes.
  Management surface: Users can review all followed volumes and quickly unfollow from a dedicated `Following` page linked from the dashboard and home rail.

- Added end-to-end browser coverage for the core `Following` happy paths.
  Coverage now proves follow/unfollow behavior, home-rail visibility, no-backfill behavior, arrival visibility after a later import, and the transition from `New from Following` into `Jump Back In` once reading starts.

- Added experimental `JPEG XL` archive support to Parker's backend image pipeline.
  Current scope: Parker now registers the Pillow JXL plugin at startup, recognizes `.jxl` pages in archives, generates thumbnails and palettes from JXL-backed archives, and preserves native `image/jxl` reader responses where the client/browser supports them.
  Release note framing: This should remain user-facing experimental support because direct reading of native `JXL` pages still depends on browser support outside Parker's control.

- Added baseline `AVIF` archive compatibility through Parker's existing Pillow-based image pipeline.
  Current scope: Parker now recognizes `.avif` pages in archives and supports thumbnail generation, palette extraction, raw page extraction, and native `image/avif` reader delivery through the same backend paths used for other supported formats.

## Changed

- Changed Parker's collection-management model so `Following` is now distinct from manual `Stacks`.
  Product note: `Stacks` remain user-curated reading buckets, while `Following` is the automatic system for surfacing later arrivals from a tracked run.

- Changed `Following` to use conservative matching rules in its first release.
  Matching rule: New arrivals are scoped to plain/mainline issues only by default.
  Queue rule: Annuals and specials are excluded in v1 rather than being inferred from inconsistent library metadata conventions.
  Exit rule: A followed arrival leaves `New from Following` once the user starts reading it or marks it read, at which point the normal resume/on-deck flows take over.

- Changed Parker's image-format handling to broaden modern archive compatibility without adding format-specific reader branches.
  Implementation note: `JPEG XL` support is enabled through one-time Pillow codec registration, while `AVIF` uses support already present in the current Pillow environment.
  Reader note: Native page responses now preserve the correct file extension and media type for both `image/jxl` and `image/avif`.

## Fixed

- Fixed brittle volume metadata parsing during import so malformed non-empty volume values fall back safely instead of failing the comic import path.
  Behavior note: Parker already treated blank volume metadata as volume `1`; it now handles invalid non-numeric values more safely as well.

- Fixed the lack of a practical unfollow workflow for users tracking many volumes.
  UX note: Users no longer need to drill back into each series and volume just to stop following a run.

- Fixed Parker's backend handling gap for archives containing newer image formats.
  `JPEG XL`: Parker can now enumerate JXL-backed archive pages, generate thumbnails and palette data from them, and serve native reader responses with `image/jxl`.
  `AVIF`: Parker can now do the same for AVIF-backed archive pages with native `image/avif` responses.
